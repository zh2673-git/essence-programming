## 哲学根基：逻辑驱动的设计

本质编程不仅仅是一种开发技术——它是一种植根于**形式逻辑**与**本体论分析**的软件工程哲学方法。

### 逻辑学基础

我们的方法论将三种基本逻辑操作应用于软件设计：

| 逻辑操作 | 软件概念 | 核心问题 |
|---------|---------|---------|
| **内涵 (Connotation)** | 本质特征 | 它本质上是什么？ |
| **外延 (Extension)** | 关键属性 | 它由什么组成？ |
| **区分 (Differentiation)** | 区分属性 | 它不是什么？ |

这三元组映射了逻辑学中的经典**定义理论**：要完整定义一个事物，必须指明其本质性质（内涵）、枚举其实例（外延）、划定其边界（区分）。

### 核心公式

```
┌─────────────────────────────────────────────────────────┐
│                    本质编程公式                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   逻辑推导：                                            │
│   ┌─────────────┐    ┌─────────────┐    ┌────────────┐ │
│   │    内涵     │ → │    外延     │ → │    区分     │ │
│   │   (是什么)   │    │  (为什么)   │    │  (怎么做)   │ │
│   └─────────────┘    └─────────────┘    └────────────┘ │
│        ↓                    ↓                  ↓        │
│   本质特征          关键属性           模块/接口         │
│   → 定义系统根本    → 推导出方法/路由  → 实现交付       │
│                                                         │
│   开发模式：                                            │
│   项目 = 类，模块 = 属性，接口 = 契约，路由 = 方法分发   │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**核心原则**：所有设计必须从**本质逻辑推导**，杜绝随意性。

## 三阶方法论

### 第1阶：是什么 —— 通过逻辑定义本质

从**形式定义理论**的视角：

```
┌─────────────────────────────────────────────────────────┐
│                   第1阶：本体论分析                      │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   本质特征（内涵）                                       │
│   └── 使系统成为其自身的必要且充分条件                   │
│                                                         │
│   关键属性（外延）                                       │
│   └── 从本质特征逻辑推导出的完整属性集合                 │
│                                                         │
│   区分属性（边界）                                       │
│   └── 将本系统与相似但不同的系统分隔开的边界             │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 第2阶：为什么 —— 因果架构

从本质推导架构决策：
- **存在理由**：每个属性为什么存在？（因果必然性）
- **技术选型**：为什么选择这个技术栈？（从属性需求推导）
- **架构设计**：为什么这样设计架构？（从关系逻辑推导）

### 第3阶：怎么做 —— 从属性实现

```
属性（静态）→ 方法（动态）→ 路由（分发）
    ↓              ↓              ↓
 模块设计      方法编排        场景处理
```

## 实战验证：Datadone 项目

> **Datadone** 是一个企业级生产数据分析系统，完全使用本质编程方法论进行架构设计。

### 项目范围

Datadone 项目展示了逻辑推导在复杂业务场景中的强大威力：

| 模块 | 业务功能 | 逻辑推导过程 |
|-----|---------|-------------|
| **数据源 (DataSource)** | 多源数据接入 | 从"数据收集"本质 → 4个数据模型（银行/通话/微信/支付宝） |
| **分析 (Analysis)** | 智能数据处理 | 从"分析"本质 → 3个分析器（银行/通话/综合） |
| **导出 (Export)** | 多格式报表生成 | 从"输出"本质 → Excel/Word 导出器 |
| **工具 (Utils)** | 支撑基础设施 | 从"系统支撑"本质 → 配置/缓存/识别模块 |

## 最佳实践

### 推荐 AI 模型

我们推荐使用 **Kimi 2.5** 以获得最佳效果。其推理能力与本质编程所需的逻辑严谨性高度契合。

### 递归深入协议

当初步的三阶分析完成后，使用以下提示词解锁递归推导的完整威力：

> **"进一步分析需深入的部分"**

这个触发器启动三阶方法论的**递归应用**：

```
初始分析
   ↓
识别 "需深入" 标记
   ↓
对每个标记属性递归执行三阶分析
   ↓
完整逻辑分解整个系统
```

模型将：
1. 识别所有标记为 "需深入"（requires in-depth analysis）的属性
2. 为每个属性生成独立的三阶文档
3. 持续递归直至达到基础类型
4. 产出完整、逻辑一致的架构

### 使用工作流

```
1. 向 AI 模型提供 SKILL.md
2. 描述你的项目/系统需求
3. 模型执行第1-3阶分析
4. 触发："进一步分析需深入的部分"
5. 模型递归分解所有复杂属性
6. 结果：完整逻辑推导的架构
```
